# Namoland Project Logic & Knowledge Base

ไฟล์นี้รวบรวม Business Logic สำคัญของฟีเจอร์ภายใน Namoland เพื่อใช้อ้างอิงการทำงานของระบบ

---

## 1. Coin Package & Valuation Logic (ระบบปรับเหรียญและมูลค่า)

เนื่องจากระบบมีการ **แอดเหรียญ (Adjust+)** เพิ่มเข้าไปในแพ็คเกจได้ ทำให้เกิดความยุ่งยากในการคำนวณ "มูลค่าต่อเหรียญที่ใช้งาน" (Recognized Revenue) ระบบจึงใช้สูตร **Original Coins (เหรียญที่ซื้อจริง)** ในการคำนวณมูลค่าทางการเงิน

### การแยกประเภทเหรียญใน 1 แพ็คเกจ
- `totalCoins` ในระบบฐานข้อมูล คือยอดเหรียญตั้งต้น (ที่ลูกค้าซื้อ) **บวก** กับเหรียญที่แอดมิน Adjust เพิ่มทีหลัง
- `origCoins` (เหรียญที่จ่ายเงินซื้อจริง) คำนวณจาก: `totalCoins - adjustUpTotal` (ยอดการถูกปรับเหรียญเพิ่มทั้งหมด)
- มูลค่าเหรียญ (Coin Rate) จะถูกคำนวณจาก: `pricePaid / origCoins`
- *หมายเหตุ: เหรียญที่ถูก Adjust เพิ่มถือว่าเป็นแพ็คเกจ/ส่วนที่ **แถมฟรี** ทำให้มีมูลค่าเป็น 0 บาท และไม่สะท้อนใน Recognized Revenue*

### กฎการคำนวณรายงานและ Dashboard (Owner Dashboard & Outstanding Coin)
1. **Recognized Revenue (รายได้ที่เกิดขึ้นจริงตอนใช้งาน):**
   - จะแยกหา `origCoins` มาตีมูลค่าต่อเหรียญ
   - **ไม่นับรวม** Transaction แบบ `ADJUSTMENT`, `EXTENSION` และ `EXPIRED` เข้ามาเป็นส่วนหนึ่งของการใช้งาน (Redeem) หรือหักรายได้
2. **Total Coins Purchased (จำนวนเหรียญที่ถูกขาย):**
   - ไม่นับรวมแพ็คเกจที่ `packageType === "ADJUSTMENT"` 
   - ถ้าในแพ็คเกจจริงมีการถูก Add เหรียญฟรีผสม ให้ลบยอดเหรียญฟรีออกเพื่อนับเฉพาะเหรียญที่ลูกค้าซื้อ
3. **Total Coins Used (จำนวนเหรียญที่ถูกเบิกใช้):**
   - นับเฉพาะรายการใช้งานจริง (เช่น จองคลาส, ยืมหนังสือ) 
   - `ไม่ถูกรวมยอด` กรณีแอดมินปรับเหรียญลดเพิ่ม หรือกรณีเหรียญหมดอายุ

## 2. Coin Package Expiration (ระบบวันหมดอายุของแพ็คเกจ)

ในกรณีที่เพิ่งสร้างแพ็คเกจใหม่ **โดยเฉพาะกรณีที่ไม่ได้ซื้อแพ็คเกจปกติ แต่แอดมินใช้ปุ่ม "เพิ่มเหรียญ" ให้เลย** ระบบจะสร้างแพ็คเกจประเภท `ADJUSTMENT` โดยมีกฏดังนี้:

- **ช่วงแรก (ยังไม่ได้ใช้):** แพ็คเกจจะ **"ยังไม่มีวันหมดอายุ"** (`firstUsedAt: null`, `expiresAt: null`)
- **First Use (การใช้งานครั้งแรก):** เมื่อลูกค้าเริ่มเอาเหรียญนี้ไปใช้งาน (จองคลาส หรือ ยืมหนังสือ) ระบบจึงจะเริ่มสตาร์ทนาฬิกานับวันหมดอายุ โดยเซ็ตค่า `firstUsedAt = <วันที่เริ่มใช้>` 
- **ระยะเวลาหมดอายุ:** ระบบบวกเวลาไปอีก **1 เดือนเต็ม (1 Month)** ตามปฏิทิน (`expiresAt = <วันที่เริ่มใช้ + 1 เดือน>`)

## 3. URLs และ Month Index (ข้อมูลเชิงเทคนิค Javascript)

ฟังก์ชันต่างๆ ที่รับค่า Parameter เป็นตัวเลขระบุเดือน (เช่น URL `/reports/outstanding/[year]/[monthIndex]`) จะประมวลผลผ่าน `Date()` ของ Javascript
- เดือนจะถูกอ้างอิงเป็นแบบ **0-Indexed**:
  - `0` หมายถึง มกราคม (January)
  - `3` หมายถึง เมษายน (April)
  - `11` หมายถึง ธันวาคม (December)
- ระบบ Backend/Report ใช้ระบบเดือน 0-11 นี้ล้วนๆ ควรอ้างอิงตามนี้เพื่อลดข้อผิดพลาดในการทำ Date Shift

## 4. Refund / Check-In Cancellation (ระบบยกเลิกคลาสแล้วคืนเหรียญ)

- **ไม่ใช่** การสร้าง Transaction ก้อนใหม่ที่เป็นบวกเพื่อคืนเงินเข้าแพ็คเกจ
- การคืนเหรียญจะใช้วิธี **"ลบ Transaction ก้อนเก่าทิ้ง (Erase)"** (`prisma.coinTransaction.delete`) และอัปเดต `remainingCoins` ให้เพิ่มกลับคืนมายังแพ็คเกจที่ถูกต้อง เพื่อให้ไม่กระทบยอด Total Coins Used ใน Dashboard รวม
